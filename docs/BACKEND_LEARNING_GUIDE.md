> 这份文档用大白话讲清楚：**后端是干什么的、我们要做成什么样、你按什么顺序学**。  
> 更细的表字段和 API 清单见：`BACKEND_DESIGN.md`。

## 0. 先建立直觉：餐厅比喻

把整个网站想成一家餐厅：

| 现实里 | 对应到项目 |
|--------|------------|
| 菜单、装修、点餐平板 | **前端**（你现在看到的 HTML 页面） |
| 厨房、收银、仓库记账 | **后端**（我们要做的程序） |
| 冰柜里的食材和订单本 | **数据库**（把用户、问卷、对局存起来） |

现在的情况是：餐厅装修好了，客人可以「假装点餐」，但厨房是空的——点了也不记、不算钱、刷新就忘。

**后端要做的事**：客人一点餐，厨房就认真做，并记到账本里；下次来还能查到。

## 1. 前端已经有了，后端还缺什么？

打开现在的页面，很多功能是「演给你看」的：

- 登录：随便填邮箱也能进（`admin@` 开头进管理端）
- 问卷：答案只在浏览器内存里，一刷新就没了
- 博弈：电脑随机装成「对方」，分数也不上传
- 排行榜 / 用户列表：写死的假数据

**正式投入使用**，必须变成：

1. 真的注册、登录、分清「普通参与者」和「管理员」
2. 问卷答案存进数据库，并算出人格五维分数
3. 每一轮博弈选择存下来，分数由服务器算（不能让浏览器自己说了算）
4. 管理端看到的是真实用户和真实结果
5. 排行榜用真实得分算出来

这些，都是后端 + 数据库的工作。

## 2. 后端到底由哪几块组成？

学后端，先记住四个词：

### ① 接口（API）

前端要数据时，会发一个网络请求，比如：

- 「我要用这个邮箱登录」
- 「帮我保存第 3 题选了 4 分」
- 「这一轮我选 A」

后端提供的这些「窗口」，就叫 **API**。  
可以想象成厨房的传菜口：前厅只通过窗口点菜，不直接进冰柜乱翻。

### ② 业务逻辑

收到请求后，后端要按规则办事，例如：

- 密码对不对？
- 44 题是不是都答完了？
- 双方都选 A 得 10 分，还是别的组合？
- 没做完问卷，能不能开博弈？

规则写在后端代码里，**不要相信前端说「我已经解锁了」**——正式环境里，前端可以被改。

### ③ 数据库

把需要长期保存的东西放进表里，例如：

- 有哪些用户
- 谁答了哪些题
- 谁在某一轮选了 A 还是 B

数据库像 Excel，但是更适合多人同时读写、能保证数据不乱。

### ④ 鉴权（身份证明）

登录成功后，后端发一张「临时门禁卡」（常见叫 **Token**）。  
之后每次请求都带上这张卡，后端才知道「你是林知夏，不是别人冒充的」。

## 3. 本项目后端要覆盖的业务（按用户故事）

用「谁要干什么」来记，比死记表名更容易：

### 参与者

1. 注册账号（邮箱 + 密码 + 昵称）
2. 登录，进入自己的空间
3. 阅读 BFI 说明，做 44 题问卷
4. 提交后看到人格结果，并解锁博弈
5. 玩猎鹿博弈两个场景，每个 10 轮
6. 两个场景都完成后，算实验完成
7. 看排行榜

### 管理员

1. 用管理员账号登录
2. 看有哪些用户、问卷有没有做完、人格分数
3. 管理实验列表（顺序、开关）
4. 看注册/登录相关设置与事件（后期再做细）

### 系统自己要保证的

- 分数按固定矩阵算，不能作弊改分
- 没问卷不能开正式博弈
- 重要操作有时间记录，方便研究复盘

## 4. 数据怎么存？先想「几张表」，别急着背字段

把数据库想成几本账：

```
用户账本 users
  └─ 谁注册了、密码（加密后）、是参与者还是管理员

问卷题库 survey_items
  └─ BFI-44 那 44 句话（以后改题不改代码）

问卷答卷 survey_responses + survey_answers
  └─ 某人某次答卷 + 每一题选了几分

人格分数 personality_scores
  └─ 算出来的 E A C N O（1～5 分）

实验与场景 experiments + experiment_scenes
  └─ 猎鹿博弈、两个场景的文案和规则

对局 game_sessions + game_rounds
  └─ 开了一局、每一轮你和对方选了什么、得了几分
```

### 一张「关系」简图（白话）

```
一个用户
  ├─ 可以有一份 BFI 答卷
  │     └─ 答卷里有 44 条答案
  │           └─ 算出一份人格分数
  └─ 可以有很多局博弈
        └─ 每一局有最多 10 条轮次记录
```

详细字段见 `BACKEND_DESIGN.md` 第 5 节。学习阶段先记住：**用户 → 答卷 → 对局** 这条主线。

## 5. 接口怎么设计？也按「办事窗口」记

统一前缀：`/api/v1`  
意思是：第一版公开接口都挂在这个路径下。

### 5.1 账号相关

| 窗口 | 干什么 |
|------|--------|
| `POST /auth/register` | 注册 |
| `POST /auth/login` | 登录，拿到 Token |
| `GET /auth/me` | 我是谁、角色是什么 |

### 5.2 问卷相关

| 窗口 | 干什么 |
|------|--------|
| `GET /surveys/bfi-44` | 拉取 44 题 |
| `PUT /surveys/bfi-44/answers` | 边做边保存（防刷新丢失） |
| `POST /surveys/bfi-44/submit` | 正式提交、计分、解锁博弈 |
| `GET /surveys/bfi-44/my-response` | 看我做到哪了 / 结果 |

### 5.3 博弈相关

| 窗口 | 干什么 |
|------|--------|
| `GET /experiments/stag-hunt/scenes` | 两个场景 + 我完成了几个 |
| `POST .../scenes/{key}/sessions` | 开始一局 |
| `POST /sessions/{id}/rounds` | 本轮我选 A 或 B（服务器算分、生成对方） |

**重要学习点：**  
前端只传「我选了什么」；「对方选什么、得几分」必须由后端算。  
这样以后别人用改网页的方式作弊，也改不了数据库里的真实记录。

### 5.4 排行榜与管理

| 窗口 | 干什么 |
|------|--------|
| `GET /leaderboard` | 排行榜 |
| `GET /admin/users` | 用户列表（仅管理员） |
| `GET /admin/users/{id}/personality` | 某人的人格详情 |
| `GET/PATCH /admin/experiments` | 实验配置 |

完整清单仍以 `BACKEND_DESIGN.md` 为准。

## 6. 学习型技术选型（推荐你就按这个学）

你是后端小白，又想靠本项目学会，建议选 **一条路走到底**，不要同时学三套。

### 推荐组合（本项目默认）

| 层级 | 选什么 | 为什么适合新手 |
|------|--------|----------------|
| 语言 | **Python** | 语法接近人话，资料多 |
| Web 框架 | **FastAPI** | 写完自动有接口文档页 `/docs`，边写边点着试 |
| 数据库 | **PostgreSQL** | 正规项目常用，和设计文档一致 |
| 访问数据库 | **SQLAlchemy** 或先学手写 SQL | 先理解「表和查询」，再加深 |
| 密码 | **密码哈希**（如 bcrypt） | 永远不要明文存密码 |
| 登录态 | **JWT Token** | 概念清晰：登录发牌，请求验牌 |

> 备选：若你更想「前后端都用 JavaScript」，可改用 Node.js + Express/Fastify + Prisma。  
> **两套都对，但学习期只选一套。** 本文后续示例默认按 Python + FastAPI 讲。

### 这些词第一次见面时怎么理解

- **框架**：别人搭好的脚手架，你往上面挂「登录、问卷」等功能，不用从零写网络服务器。
- **迁移（migration）**：用文件记录「数据库表怎么建、怎么改」，团队/以后自己能重复执行，而不是手点数据库软件。
- **环境变量**：把数据库密码、密钥放在配置里，不写死在代码中。
- **种子数据（seed）**：预先插入管理员账号、44 道题，方便本地一键开始测。

## 7. 推荐的项目目录（以后创建时）

```
yangmind/
├── docs/                    # 设计文档（现在就有）
├── index.html …             # 现有前端原型（先别动）
└── backend/                 # 后端（实现阶段再建）
    ├── app/
    │   ├── main.py          # 程序入口
    │   ├── api/             # 各个接口（auth、survey、game…）
    │   ├── models/          # 数据库表对应的模型
    │   ├── services/        # 业务规则（计分、解锁判断）
    │   └── core/            # 配置、安全、数据库连接
    ├── migrations/          # 建表脚本
    ├── tests/               # 测试（学会后再加）
    ├── requirements.txt     # Python 依赖列表
    └── README.md            # 怎么启动后端
```

**学习口诀：**  
`api` 负责「收请求、回结果」；`services` 负责「按规则办事」；`models` 负责「和表对应」。

## 8. 你怎么边做边学？（学习路线图）

别一上来就想「把整个平台写完」。按关卡打，每关都有可运行的成果。

### 第 0 关：环境（1～2 天）

- 安装 Python、PostgreSQL（或用 Docker 一键起数据库）
- 跑通一个「Hello」接口：浏览器打开能看到 `{"ok": true}`
- **你学到：** 后端是一个一直开着的程序，等别人来请求

### 第 1 关：用户注册登录（核心基本功）

- 建 `users` 表
- 注册：密码哈希后入库
- 登录：校验密码，返回 JWT
- 做一个「需要登录才能访问」的 `/auth/me`
- **你学到：** 鉴权、哈希、Token、什么叫受保护接口

### 第 2 关：问卷

- 题库种子（44 题）
- 保存答案、提交、算五维分
- 规则：没提交完不能算正式完成
- **你学到：** 一对多表关系、事务（要么全成功要么全别半吊子）

### 第 3 关：猎鹿博弈

- 开局、每轮只收 choice
- 服务器生成对方选择 + 按矩阵计分
- 两个场景完成判定
- **你学到：** 服务端权威、状态机（进行中 / 已结束）

### 第 4 关：管理端与排行榜

- 管理员权限判断
- 用户列表、人格详情
- 排行榜聚合查询（`SUM`、排序）
- **你学到：** 角色权限、统计查询

### 第 5 关：和正式前端对接

- 那时再上 React 或给旧页面接 API
- **你学到：** 前后端分离真正通了

每关结束后问自己三句话：

1. 数据存在哪张表？  
2. 谁可以调用这个接口？  
3. 坏人改前端，能不能骗过后端？

## 9. 本期已经定下来的设计决策（减少纠结）

| 问题 | 决定 | 白话解释 |
|------|------|----------|
| 先前端还是先后端？ | **先后端** | 正式站用 React 也要接同一套接口 |
| 现有 HTML？ | **先当原型，不改** | 需求说明书 |
| MVP 对手怎么来？ | **机器人策略（bot）** | 先学会存对局；真人匹配以后再加 |
| 囚徒困境？ | **表先预留，逻辑可后做** | 别第一期摊太大 |
| 问卷能否反复交？ | **默认交一次；要重测找管理员重置** | 研究数据更干净 |

## 10. 最小可行后端（MVP）成功标准

做到下面这些，你就已经「会做一个能用的后端雏形」了：

1. 能注册、登录，Token 有效  
2. 能答完 BFI 并看到五维分数，数据在数据库里刷新还在  
3. 能打完猎鹿两个场景，每一轮都有记录，分数服务器算的  
4. 管理员能看到用户列表  
5. 现有 HTML 演示仍可单独打开，互不干扰

## 11. 你接下来可以怎么用我

学习项目里，比较高效的协作方式是：

1. **你说「开始第 0 关」** → 我带你建 `backend/` 目录、装依赖、跑第一个接口（边做边讲）  
2. 每一关只做一个小目标，做完你用 `/docs` 自己点一遍  
3. 遇到报错把终端信息贴给我，一起查（查错也是后端基本功）

不建议：一口气让我「生成全部后端」你却看不懂——那样学不到。

## 12. 和正式设计文档的关系

| 文档 | 用途 |
|------|------|
| `BACKEND_LEARNING_GUIDE.md`（本文） | 白话、学习顺序、整体设计直觉 |
